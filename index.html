trifl

a functional web client user interface library with a unidirectional dataflow and a virtual dom.

Trifl provides some simple functions to structure web applications. There are actions to aid a unidirectional data flow, views to visualize your model state as it changes and router to glue the url to actions and views.

Discuss trifl on gitter, read the api docs, review the code on github and report issues on github.

no async

Trifl does not have callback or asynchronous methods. Everything from view rendering to routing is done synchronously.

think big

Generally with trifl, it is encouraged to "think big", big actions, big model changes, big view updates. Trifl provides the tools to avoid micro managing model/dom updates.

three parts

There are three parts to trifl:

  • Actions are helper functions to aid decoupling of application parts in a unidirectional manner.
  • Views are render functions whose purpose are to make dom nodes reflect some model state.
  • The router is a utility for organising a url space into visible views and firing actions as a result of url changes.

actions

Actions are triggered by input or asynchronous server events such as responses to ajax.

Actions can be thought of as really bad events cause they can only have one listener: the handler. This is one part of "thinking big". Traditional events encourage micro state changes with multiple listeners scattered througout the code. In medium to large applications it quickly become very hard to get an overview of all things happening as a result of such an event.

In trifl there is only one action handler per action, and, in line with unidirectional thinking, its only responsibility is to propagate the action to the model.

Other frameworks talk about "dispatcher" as a component, and whilst trifl doesn't have an explicit such an component, we encourage to think of action handlers as dispatcher code.

Declare a dispatcher function

handler "selectarticle", (articleid) ->
    # action updates the model
    model.articles.requestArticle(articleid)

Trigger an action

action "selectarticle", "slugattack01"

Declare a dispatcher function

handler("selectarticle", function(articleid) {
    // action updates the model
    model.articles.requestArticle(articleid);
});

Trigger an action

action("selectarticle", "slugattack01");

Only one action can be triggered at a time. It is an error to dispatch another somewhere inside the handler (or model code executed by the handler) and doing so will result in an exception.

There is however another class of actions called updates that are allowed. Updates are used to signal that a model has been updated and requires re-rendering in the views.

dispatchers and controllers

Handlers come in two variants that work exactly the same, but are mentally separate to know what part of the code they belong in.

  • action – dispatcher function (handler)
  • update – controller function (handler)

Dispatchers are handlers that receive plain actions. Controllers work on the back of models marked as updated during the dispatch of an action.

Declare a controller function

handler "update:articles", ->
    # rerender views
    views.articleCount model.articles
    views.articleView model.articles

Trigger an update (as part of an action)

model.articles = {
  requestArticle: (articleid) =>
    url = "/get/#{articleid}"
    $.ajax(url).done (article) ->
      # trigger action with the update
      action "gotarticle", article 
    @state = "requesting"
    update "articles"
}

Declare a controller function

handler("selectarticle", function(articleid) {
    // render views
    views.articleCount(model.articles);
    views.articleView(model.articles);
});

Trigger an update (as part of an update)

model.articles = {
  requestArticle: function(articleid) {
    var url = "/get" + articleid";
    var _this = this;
    $.ajax(url).done(function(data) {
      // trigger action with the update
      action("gotarticle", article);
    });
    this.state = "requesting";
    update("articles");
  }
};

views

Views render a model state into something the user can see. Trifl ships with tagg, configured to deep hook into virtual-dom.

In views we again "think big" and encourage to render large parts of the screen – entire models – at a time. The virtual dom will figure out the differences, and only make partial real dom changes.

Views are render-functions that are wrapped to provide the virtual-dom-goodness. Using the awesomeness of javascript, we set a property on the created function, view.el. This is the dom element the view function controls.

Initially, before the view function has ever been invoked, this is a simple <div></div>, but the type can change on first function invokation.

Create a view function

articleview = view (articles) ->
  ul ->
    articles.forEach (article) ->
      li article.title, ->
        span class:'desc', article.description

# articleview.el is now a DOM node.

Render it

articleview model.articles

Create a view function

var articleview = view(function(articles) {
  ul(function() {
    articles.forEach(function(article) {
      li(article.title, function() {
        span({className:'desc'},
          article.description);
      });
    });
  });
});

// articleview.el is now DOM node.

Render it

articleview(model.articles);

layouts

Layouts are special views, they provide a means of organizing the dom into named "pigeon holes", called regions, that display other views.

Layout functions typically don't take arguments and are always invoked upon creation.

Create a layout function

applayout = layout ->
  div ->
    div region('topnav')
    div region('leftnav')
    div region('main')

# applayout.el is now the
# outer div

# applayout.topnav,
# applayout.leftnav
# applayout.main are region
# functions

Put a view in a region

applayout.main articleview

Create a layout function

applayout = layout(function() {
  div(function() {
    div(region('topnav'));
    div(region('lefnav'));
    div(region('main'));
  });
});

// applayout.el is now the
// outer div

// applayout.topnav,
// applayout.leftnav
// applayout.main are region
// functions

Put a view in a region

applayout.main(articleview);

router

The router turns the url space into executable code. There is only one route function which use nested path statements to "consume" the current url from left to right.

The route is always executed straight away when set.

Declare the route function

route ->

    # starting from the left,
    # we consume "/contact"
    path '/contact', ->

        # if current url starts
        # "/contact", this block
        # is executed

        # put contact view in
        # main region
        appview.main contactview

Nested path functions

route ->

    path '/aboutus', ->

        appview.main aboutusview

        path '/history', ->

            # we only execute
            # this block if
            # current url starts
            # /aboutus/history

Declare the route function

route(function() {

    // starting from the left,
    // we consume "/contact"
    path('/contact', function() {

        // if current url starts
        // "/contact", this block
        // is executed

        // put contact view in
        // main region
        appview.main(contactview);
    });
});

Nested path functions

route(function() {

    path('/aboutus', function() {

        appview.main(aboutusview);

        path('/history', function() {

            // we only execute
            // this block if
            // current url starts
            // /aboutus/history

        });
    });
});

exec

To do something useful with the remainder, the not consumed part of the url, we use exec. This executes the given function with two arguments: the remainder and the query object.

Execute from the current context.

# current url is /news/panda?format=json

route ->

    path '/news/', ->

        exec (remainder, query) ->

            # remainder is "panda"
            # query is {format:'json'}

Execute from the current context.

// current url is /news/panda?format=json

route(function() {

    path('/news/', function() {

        exec(function(remainder, query) {

            // remainder is "panda"
            // query is {format:'json'}

        });
    });
});

navigate

navigate does a pushState change of the url. This triggers the route function if the new url differs from the previous.

Change the url.

navigate '/some/where?q=apples'

Change the url.

navigate('/some/where?q=apples');
